# Data Modeling in Django

This page covers how to define each relationship type in `models.py` and apply the resulting migrations. For theory on when to use each type, see [Types of Relationships](/type).

---

## Defining Relationship Fields

<Tabs items={['One To One', 'One To Many', 'Many To Many']}>
    <Tabs.Tab>
        ### One To One

        Use `OneToOneField` when each record in one model maps to exactly one record in another. Place it on whichever model is the natural "owner" of the relationship.

        ```python
        # models.py
        from django.db import models

        class Address(models.Model):
            street = models.CharField(max_length=80)
            postal_code = models.CharField(max_length=5)
            city = models.CharField(max_length=50)

            def __str__(self):
                return f"{self.street}, {self.postal_code} {self.city}"

        class Meta:
            verbose_name_plural = "address entries"

        class Author(models.Model):
            first_name = models.CharField(max_length=100)
            last_name = models.CharField(max_length=100)
            address = models.OneToOneField(
                Address,
                on_delete=models.CASCADE,
                null=True,
            )

            def __str__(self):
                return f"{self.first_name} {self.last_name}"
        ```

        <Callout>
            The reverse accessor for a `OneToOneField` is singular — `address.author` — not a set, because only one record can ever match.
        </Callout>
    </Tabs.Tab>

    <Tabs.Tab>
        ### One-to-Many(ForeignKey)

        Place the `ForeignKey` on the model that holds the reference — the "many" side. Always define the referenced model (`Author`) above the referencing model (`Book`) in the file, or use a string reference.

        ```python
        # models.py
        from django.db import models

        class Author(models.Model):
            first_name = models.CharField(max_length=100)
            last_name = models.CharField(max_length=100)

            def __str__(self):
                return f"{self.first_name} {self.last_name}"

        class Book(models.Model):
            title = models.CharField(max_length=200)
            author = models.ForeignKey(
                Author,
                on_delete=models.CASCADE,
                related_name="books",
                null=True,
            )

            def __str__(self):
                return self.title
        ```

        <Callout type="info">
            `null=True` is required when adding a `ForeignKey` to a model that already has rows — it lets Django set existing records to `NULL` during migration instead of requiring a one-off default.
        </Callout>

        #### `on_delete` options

        | Option | Behavior |
        |---|---|
        | `CASCADE` | Deletes all related books when the author is deleted |
        | `SET_NULL` | Sets the field to `NULL` (requires `null=True`) |
        | `SET_DEFAULT` | Sets the field to its default value |
        | `PROTECT` | Prevents deletion if related records exist |
        | `DO_NOTHING` | Takes no action (use with caution) |
    </Tabs.Tab>

    <Tabs.Tab>
        ### Many To Many

        Define the field on whichever model feels most natural. Django automatically creates a hidden intermediate table to store the mapping — you never define or migrate it manually.

        ```python
        # models.py
        from django.db import models

        class Country(models.Model):
            name = models.CharField(max_length=80)
            code = models.CharField(max_length=2)

            def __str__(self):
                return self.name

        class Meta:
            verbose_name_plural = "Countries"

        class Book(models.Model):
            title = models.CharField(max_length=200)
            published_countries = models.ManyToManyField(
                Country,
                related_name="books",
            )

            def __str__(self):
                return self.title
        ```

        <Callout type="warning">
            Do **not** add `on_delete` to a `ManyToManyField`. Django manages deletions through the intermediate table automatically.
        </Callout>
    </Tabs.Tab>
</Tabs>

## Applying Migrations

After defining or changing any model field, run:

```bash
python manage.py makemigrations
python manage.py migrate
```

### Handling migration warnings

When changing a field type (e.g., from a plain `CharField` to a `ForeignKey`), Django warns that the field is non-nullable without a default. This is because the old column stored text while the new column expects an integer ID.

**Fix:** Add `null=True` to the field before running `makemigrations` again.

### Handling invalid foreign key errors

If you see:

```
Row in table 'bookoutlet_book' with primary key 2 has an invalid foreign key.
```

Existing rows contain old text values in a field that now expects integer IDs. Clear the stale data before migrating:

```bash
python manage.py shell
```

```python
from bookoutlet.models import Book
Book.objects.all().delete()
exit()
```

Then re-run `migrate`.

<Callout type="info">
    Design your models before adding data. Changing field types on a populated table requires either clearing that data or writing a custom data migration.
</Callout>
