# Django Forms & Class-Based Views

A complete guide to handling user input with Django ŌĆö from raw HTML forms to ModelForms and class-based views.

---

## Why Forms Matter

The Django admin panel is great for you as the developer, but it is not meant for site visitors. When you need users to submit data ŌĆö reviews, comments, feedback ŌĆö you need **user-facing forms**.

This guide builds a **review submission feature** from scratch. By the end you will understand:

- How GET and POST requests differ
- What CSRF protection is and why it is non-negotiable
- How to use Django's `Form` class to replace tedious manual validation
- How `ModelForm` connects a form directly to a database model
- How class-based views clean up the `if request.method == 'POST'` pattern

## Part 2: A Raw HTML Form

Before using any Django form tooling, here is what a plain HTML form looks like and what each piece does.

Add this to the `<body>` of `reviews/templates/reviews/review.html`:

```html filename="reviews/templates/reviews/review.html"
<form method="post">
    {% csrf_token %}
    <label for="username">Your Name</label>
    <input type="text" id="username" name="username" />
    <button type="submit">Send</button>
</form>
```

Three things to notice:

**`action="/"`** tells the browser where to send the submitted data. An empty `action` or `action="/"` submits back to the same URL.

**`method="post"`** sends data in the request body instead of appending it to the URL as `?username=Max`. Use `POST` whenever you are creating or changing data.

**`{% csrf_token %}`** is a Django template tag that injects a hidden security token. Without it, Django refuses the submission with a 403 Forbidden error.

<Callout type="error">
    **Never omit `{% csrf_token %}`** from a POST form. Django's CSRF middleware rejects every POST that lacks a valid token. This protects your users from Cross-Site Request Forgery attacks, where a malicious page tricks a browser into submitting a request to your site without the user's knowledge.
</Callout>

### Reading submitted data in the view

Update `reviews/views.py` to handle the POST:

```python filename="reviews/views.py"
from django.shortcuts import render
from django.http import HttpResponseRedirect

def review(request):
    if request.method == 'POST':
        entered_username = request.POST['username']  # matches the input's name=""
        print(entered_username)
        return HttpResponseRedirect('/reviews/thank-you')
    return render(request, 'reviews/review.html')

```

`request.POST` is a dictionary. The key is the `name` attribute from your `<input>` ŌĆö so `name="username"` maps to `request.POST['username']`.

    <Callout type="info">
        **Always redirect after a successful POST.** If you render a template directly after saving data, refreshing the page re-submits the form. Redirecting to a new URL breaks that cycle ŌĆö this pattern is called **Post/Redirect/Get**.
    </Callout>

    ### Why raw HTML forms stop scaling

    This approach works for a single field, but as soon as you add more fields you need to manually validate every one:

    ```python filename="reviews/views.py"
    # ŌØī This gets unwieldy fast
    if request.method == 'POST':
        username = request.POST['username']
    if not username:
        return render(request, 'reviews/review.html', {'has_error': True})
    if len(username) > 100:
        return render(request, 'reviews/review.html', {'has_error': True})
    # ... and so on for every field
    ```

    A form with five fields and three rules each means fifteen `if` blocks ŌĆö all written by hand, all repeated across every form in your project. Django's `Form` class eliminates this entirely.

    ---

    ## Part 3: The Django `Form` Class

    ### Define the form

    Create `reviews/forms.py` (this file does not exist yet):

    ```python filename="reviews/forms.py"
    from django import forms

    class ReviewForm(forms.Form):
        user_name = forms.CharField(
            label='Your Name',
            max_length=100,
            error_messages={
                'required': 'Your name must not be empty.',
                'max_length': 'Please enter a shorter name.',
            }
        )

        review_text = forms.CharField(
            label='Your Feedback',
            max_length=200,
            widget=forms.Textarea,   # renders a <textarea> instead of <input type="text">
        )

        rating = forms.IntegerField(
            label='Your Rating',
            min_value=1,
            max_value=5,
        )
```

### Use the form in the view

Update `reviews/views.py`:

```python filename="reviews/views.py"
from django.http import HttpResponseRedirect
from django.shortcuts import render

from .forms import ReviewForm

def review(request):
    if request.method == 'POST':
        form = ReviewForm(request.POST)
        if form.is_valid():
            print(form.cleaned_data)
        return HttpResponseRedirect('/reviews/thank-you')
    else:
        form = ReviewForm()
    return render(request, 'reviews/review.html', {'form': form})

def thank_you(request):
    return render(request, 'reviews/thank_you.html')
```

<Callout>
  `form.is_valid()` does three things at once: validates all fields, returns `True` or `False`, and populates `form.cleaned_data` with sanitised values (only when valid). You never need to write the validation logic yourself.
</Callout>

### Render the form in the template

Update `reviews/templates/reviews/review.html`. Replace the manual `<input>` with Django's form object:

```html filename="reviews/templates/reviews/review.html"
<form method="post">
    {% csrf_token %}
    {{ form }}
    <button type="submit">Send</button>
</form>
```

`{{ form }}` expands to all the `<label>` and `<input>` elements Django knows about from your `ReviewForm` definition. It also re-populates fields with the user's previous input and injects error messages automatically when validation fails.

## Part 4: Three Ways to Render a Form

All three approaches go in the **same place** ŌĆö inside the `<form>` tag in `reviews/templates/reviews/review.html`, replacing `{{ form }}`. They differ only in how much control you want over the surrounding HTML.

<Tabs items={['Option 1 ŌĆö {{ form }}', 'Option 2 ŌĆö {{ form.as_p }}', 'Option 3 ŌĆö {% for field in form %}']}>
  <Tabs.Tab>

Renders every field as a flat sequence of label/input pairs. Good for prototyping.

```html filename="reviews/templates/reviews/review.html"
<form method="post">
  {% csrf_token %}
    {{ form }}
    <button type="submit">Send</button>
</form>
```

Django outputs something like:

```html
<label for="id_user_name">Your Name:</label>
<input type="text" name="user_name" maxlength="100" required id="id_user_name">
<!-- error messages appear here when invalid -->
    <label for="id_review_text">Your Feedback:</label>
<textarea name="review_text" ...></textarea>
    <!-- ... -->
    ```

</Tabs.Tab>
<Tabs.Tab>

    Each field is wrapped in a `<p>` tag automatically ŌĆö a small step up in readability with no extra code.

    ```html filename="reviews/templates/reviews/review.html"
    <form method="post">
        {% csrf_token %}
        {{ form.as_p }}
        <button type="submit">Send</button>
    </form>
    ```

    Use `{{ form.as_ul }}` or `{{ form.as_table }}` for list and table wrappers respectively.

</Tabs.Tab>
<Tabs.Tab>

    Loop over each field individually. This is what you use in production when you need custom CSS classes, error styling, or a specific DOM structure.

    ```html filename="reviews/templates/reviews/review.html"
    <form method="post">
        {% csrf_token %}

        {% for field in form %}
        <div class="form-control {% if field.errors %}errors{% endif %}">
            {{ field.label_tag }}
            {{ field }}
            {{ field.errors }}
        </div>
        {% endfor %}

        <button type="submit">Send</button>
    </form>
    ```

    What each part renders:

    | Tag | Output |
    |---|---|
    | `field.label_tag` | The `<label>` element |
    | `field` | The `<input>`, `<textarea>`, or `<select>` |
| `field.errors` | A `<ul>` of error messages ŌĆö empty when the field is valid |
| `{% if field.errors %}errors{% endif %}` | Adds an `errors` CSS class to the wrapper `<div>` so you can style invalid fields |

<Callout type="info">
  **When to use each option:** use `{{ form }}` while prototyping. Switch to the `{% for field in form %}` loop once you are applying real CSS or need specific markup required by your design.
</Callout>

</Tabs.Tab>
</Tabs>

## Part 5: Saving to a Database

### Define the model

```python filename="reviews/models.py"
from django.db import models

class Review(models.Model):
    user_name   = models.CharField(max_length=100)
    review_text = models.TextField()
    rating      = models.IntegerField()
```

### Run migrations

```bash
python manage.py makemigrations
python manage.py migrate
```

### Save validated form data

Update the `review` view in `reviews/views.py`:

```python filename="reviews/views.py"

from django.shortcuts import render
from django.http import HttpResponseRedirect
from .forms import ReviewForm
from .models import Review

def review(request):
    if request.method == 'POST':
        form = ReviewForm(request.POST)
    if form.is_valid():
        rev = Review(
            user_name=form.cleaned_data['user_name'],
            review_text=form.cleaned_data['review_text'],
            rating=form.cleaned_data['rating'],
        )
        rev.save()
        return HttpResponseRedirect('/reviews/thank-you')
    else:
        form = ReviewForm()

    return render(request, 'reviews/review.html', {'form': form})

def thank_you(request):
    return render(request, 'reviews/thank_you.html')

```

<Callout>
    **Form fields vs. model fields** ŌĆö they look similar but serve different purposes. `forms.CharField` controls how data is captured and validated in the browser. `models.CharField` controls how data is stored in the database. A `ModelForm` (next part) links the two automatically.
</Callout>

## Part 6: ModelForm ŌĆö Skipping the Repetition

When your form maps closely to a model, `ModelForm` infers the form controls from the model definition ŌĆö you no longer declare fields twice.

### Replace `forms.Form` with `forms.ModelForm`

Update `reviews/forms.py`:

```python filename="reviews/forms.py"
from django import forms
from .models import Review

class ReviewForm(forms.ModelForm):
    class Meta:
        model = Review
        fields = '__all__'  # include every model field
        labels ={
            'user_name': 'Your Name',
            'review_text': 'Your Feedback',
            'rating': 'Your Rating',
        }
        error_messages = {
            'user_name': {
                'required': 'Your name must not be empty.',
                'max_length': 'Please enter a shorter name.',
            }
        }
```

<Tabs items={["fields = '__all__'", 'Explicit list', 'Exclude']}>
    <Tabs.Tab>

        Include every field from the model (except the auto-generated `id`):

        ```python
        fields = '__all__'
        ```

    </Tabs.Tab>
    <Tabs.Tab>

        Include only specific fields ŌĆö useful when the model has fields the user should not edit:

        ```python
        fields = ['user_name', 'review_text', 'rating']
        ```

    </Tabs.Tab>
    <Tabs.Tab>

        Include everything except certain fields:

        ```python
        exclude = ['owner_comment']
        ```

    </Tabs.Tab>
</Tabs>

### Simplify the view with `form.save()`

Because `ReviewForm` is now connected to `Review`, you can replace the manual object construction with a single call.

Update `reviews/views.py`:

```python filename="reviews/views.py"
from django.shortcuts import render
from django.http import HttpResponseRedirect
from .forms import ReviewForm

def review(request):
    if request.method == 'POST':
        form = ReviewForm(request.POST)
        if form.is_valid():
            form.save()
            return HttpResponseRedirect('/reviews/thank-you')
    else:
        form = ReviewForm()

    return render(request, 'reviews/review.html', {'form': form})
```

**Updating an existing record** works the same way ŌĆö pass `instance`:

```python filename="reviews/views.py"
def edit_review(request, review_id):
    existing_review = Review.objects.get(id=review_id)

    if request.method == 'POST':
        form = ReviewForm(request.POST, instance=existing_review)
        if form.is_valid():
            form.save()
            return HttpResponseRedirect('/reviews/thank-you')
    else:
        form = ReviewForm(instance=existing_review)

    return render(request, 'reviews/review.html', {'form': form})
```

The template stays exactly the same ŌĆö no changes needed there when switching from `Form` to `ModelForm`.

## Part 7: Class-Based Views

Every function-based view so far starts with `if request.method == 'POST'`. Class-based views replace that guard with dedicated methods named after HTTP verbs ŌĆö `get()` handles GET requests, `post()` handles POST requests.

<Callout type="info">
    Class-based views are optional. You can freely mix them with function-based views in the same project. They are most useful when a view handles multiple HTTP methods with distinct logic ŌĆö exactly the form pattern you have been building.
</Callout>

### Convert the review view

Replace the `review` function in `reviews/views.py` with a class:

```python filename="reviews/views.py"
from django.views import View
from django.shortcuts import render
from django.http import HttpResponseRedirect
from .forms import ReviewForm
from .models import Review

class ReviewView(View):
    def get(self, request):
        form = ReviewForm()
        return render(request, 'reviews/review.html', {'form': form})

    def post(self, request):
        form = ReviewForm(request.POST)
        if form.is_valid():
            form.save()
            return HttpResponseRedirect('/reviews/thank-you')
        return render(request, 'reviews/review.html', {'form': form})

class EditReviewView(View):
    def get(self, request, review_id):
        existing_review = Review.objects.get(id=review_id)
        form = ReviewForm(instance=existing_review)
        return render(request, 'reviews/review.html', {'form': form})

    def post(self, request, review_id):
        existing_review = Review.objects.get(id=review_id)
        form = ReviewForm(request.POST, instance=existing_review)
        if form.is_valid():
            form.save()
            return HttpResponseRedirect('/reviews/thank-you')
        return render(request, 'reviews/review.html', {'form': form})

def thank_you(request):
    return render(request, 'reviews/thank_you.html')
```

### Register the class-based view
reviews/urls.py ŌĆö note the .as_view() call, which is required for class-based views:

```python filename="reviews/urls.py"
urlpatterns = [
    path('', views.ReviewView.as_view(), name='review'),
    path('thank-you', views.thank_you, name='thank-you'),
    path('<int:review_id>/edit', views.EditReviewView.as_view(), name='edit-review'),
]
```

The logic is identical to the function-based version ŌĆö it is simply split across two methods instead of branching with `if`.

<Callout type="warning">
    Always call `.as_view()` when registering a class-based view in `urlpatterns`. Without it, Django cannot route incoming requests to the correct method handler and will raise an error.
</Callout>

### Built-in specialised view classes

Django ships with many view classes that go further than the generic `View`:

| Class | What it does |
|---|---|
| `View` | Generic base ŌĆö define `get`, `post`, etc. yourself |
| `ListView` | Queries and renders a list of model objects |
| `DetailView` | Queries and renders a single model object |
| `FormView` | Handles the full form lifecycle with less boilerplate |
| `CreateView` | Creates a new model object from a form |
| `UpdateView` | Updates an existing model object from a form |
| `DeleteView` | Deletes a model object |

See the [Django docs ŌåÆ Built-in class-based views](https://docs.djangoproject.com/en/stable/ref/class-based-views/) for the full reference.

## Summary

<Callout>
    **The progression this guide followed:**

    1. **Raw HTML form** ŌĆö works, but reading `request.POST` by hand and writing manual validation does not scale.
    2. **`forms.Form`** ŌĆö declare fields once; Django handles rendering, validation, error messages, and data re-population automatically.
    3. **`forms.ModelForm`** ŌĆö infers form controls from a model definition; `form.save()` replaces manual object construction.
    4. **Class-based views** ŌĆö replace `if request.method == 'POST'` with dedicated `get()` and `post()` methods, keeping each handler focused.

    Two rules that apply everywhere: always include `{% csrf_token %}` in every POST form, and always redirect after a successful POST.
</Callout>
