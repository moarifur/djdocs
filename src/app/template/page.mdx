# Django Templates & Static Files

## Overview

Up to this point, you've learned how to work with URLs and views in Django and built a basic website. While functional, returning raw HTML strings from view functions introduces several maintainability problems — this guide addresses all of them.

The current approach produces output that is not visually structured, lacks proper HTML document semantics, and doesn't scale for real-world applications.

---

## Problems with Inline HTML in Views

When constructing HTML directly inside view functions, you end up with code like this:

```python
return HttpResponse("<ul><li>...</li></ul>")
```

This approach has three core problems:

**Poor HTML structure** — The output lacks key structural elements like `<!DOCTYPE>`, `<head>`, and metadata needed for SEO.

**Hard to maintain** — Constructing complex HTML inside Python strings is impractical. Imagine building an e-commerce page with navigation, products, and a footer this way.

**Mixing logic with presentation** — Markup and Python logic are tightly coupled, making both harder to read and modify.

---

## What This Guide Covers

This guide introduces two core Django features:

- **Templates** — Standalone HTML files with dynamic content support via the Django Template Language (DTL)
- **Static files** — CSS, JavaScript, and images to style your site

---

## Part 1: What Are Templates?

A **template** is a standard HTML file that holds markup and can also render dynamic content. Instead of building HTML tags inside `views.py`, you move presentation logic into dedicated HTML documents.

This follows the **Separation of Concerns (SoC)** principle — keeping logic and presentation in separate layers.

---

## Part 2: Template Folder Structure

Django follows a strict convention for where template files should live.

### Recommended Structure

<FileTree>
    <FileTree.Folder name="project_folder" defaultOpen>
        <FileTree.Folder name="app_folder (e.g. challenges)" defaultOpen>
            <FileTree.Folder name="templates" defaultOpen>
                <FileTree.Folder name="app_folder (e.g. challenges)" defaultOpen>
                    <FileTree.File name="challenge.html" />
                </FileTree.Folder>
            </FileTree.Folder>
        </FileTree.Folder>
    </FileTree.Folder>
</FileTree>

### Why the Nested Folder?

The inner `challenges/` folder inside `templates/` exists to **prevent naming conflicts**. When Django discovers templates from multiple apps, namespacing them under the app name avoids ambiguity.

For example, two apps might both have an `index.html`. The nested folder makes them distinguishable as `app_one/index.html` and `app_two/index.html`.

## Part 4: Rendering Templates in Views

Django provides `render_to_string` — a utility function that reads an HTML template file and converts its contents to a string, which you then send back as an HTTP response.

<Steps>

    ### Import the required functions

    ```python filename="challenges/views.py"
    from django.template.loader import render_to_string
    from django.http import HttpResponse
    ```

    ### Use `render_to_string` in your view

    ```python filename="challenges/views.py"
    def monthly_challenge(request, month):
    # ... your logic here
    response_data = render_to_string("challenges/challenge.html")
    return HttpResponse(response_data)
    ```

</Steps>

<Callout type="warning">
    Always reference the template using the `<app_name>/<filename>` path — for example, `"challenges/challenge.html"` — not just the filename alone. This is precisely why the nested folder structure exists.
</Callout>

## Part 5: Configuring Django to Find Templates

Django's template discovery is configured in `settings.py` under the `TEMPLATES` list. Two key options control how Django locates template files:

| Option | Description |
|--------|-------------|
| `DIRS` | A list of absolute directory paths Django should search for templates |
| `APP_DIRS` | When `True`, Django automatically searches for a `templates/` folder inside each registered app |

<Tabs items={['APP_DIRS (Recommended)', 'DIRS (Manual)']}>
    <Tabs.Tab>

        Keep `APP_DIRS: True` (the default) and register your app in `INSTALLED_APPS`. Django will then automatically discover the `templates/` folder inside any registered app.

        ```python filename="settings.py"
        TEMPLATES = [
        {
            ...
                'DIRS': [],        # Keep empty
            'APP_DIRS': True,  # Already True by default
            ...
        }
        ]
        ```

    </Tabs.Tab>
    <Tabs.Tab>

        You can manually point Django to your template folder using `DIRS`:

        ```python filename="settings.py"
        TEMPLATES = [
        {
            ...
                'DIRS': [BASE_DIR / 'challenges' / 'templates'],
            'APP_DIRS': False,
            ...
        }
        ]
        ```

        `BASE_DIR` is already defined in `settings.py` and points to your project root. Sub-paths are built using the `/` operator (`pathlib` syntax).

        <Callout type="error">
            **Downside:** You must update `DIRS` every time you add a new app. This approach does not scale well and is not recommended for app-specific templates.
        </Callout>

    </Tabs.Tab>
</Tabs>

## Part 6: Registering Your App

For `APP_DIRS` to work, your app must be registered in `INSTALLED_APPS`. Without this step, Django is not fully aware of your app and will not scan it for templates.

```python filename="settings.py"
INSTALLED_APPS = [
'django.contrib.admin',
'django.contrib.auth',
'django.contrib.contenttypes',
'django.contrib.sessions',
'django.contrib.messages',
'django.contrib.staticfiles',
'challenges',  # ← Add your app here
]
```

<Callout>
    The string must match your **app folder name** and the `name` attribute defined in your app's `apps.py` file.
</Callout>

Once registered, Django will look inside `challenges/templates/` automatically — no need to manually configure `DIRS`.

The built-in entries (like `admin`, `auth`, `contenttypes`) are Django's own internal modules. This reinforces that apps don't have to be large standalone products — they are modular units of a project.

---

## Part 7: How Template Discovery Works

When you call `render_to_string("challenges/challenge.html")`, Django executes the following steps:

1. Loops through all apps listed in `INSTALLED_APPS`
2. For each app, checks whether a `templates/` folder exists
3. Within those folders, searches for the path `challenges/challenge.html`
4. Returns the first match found

<Callout type="info">
    If no match is found, Django raises a `TemplateDoesNotExist` error and shows a list of all directories it searched — a useful debugging tool.
</Callout>

---

## Part 8: Global vs. App-Specific Templates

| Template Type | Where to Store | How to Configure |
|---------------|----------------|-----------------|
| App-specific | `<app>/templates/<app>/` | Use `APP_DIRS: True` + register app |
    | Global (shared across apps) | Any folder outside an app | Add path to `DIRS` in `settings.py` |

    For global templates used by multiple apps — such as a shared base layout — add the folder path to `DIRS`. For everything else, the `APP_DIRS` approach is best practice: it keeps your settings lean and your project modular.

    ## Part 9: The Full Request-to-Response Flow

    ```
    User hits a URL
    ↓
    Django calls the view
    ↓
    View calls render_to_string()
    ↓
    Django locates the template file
    ↓
    Template content returned as a string
    ↓
    HttpResponse sends HTML to browser
    ↓
    Browser renders the page
    ```

    ## Part 10: Debugging — The `TemplateDoesNotExist` Mystery

    <Callout type="warning" emoji="🐛">
        This is a common stumbling block. Read carefully — it can save hours of debugging.
    </Callout>

    While integrating templates, everything appeared correctly configured:

    - The template was created at `challenges/templates/challenges/challenge.html`
    - The `challenges` app was added to `INSTALLED_APPS`
    - The view used `render_to_string("challenges/challenge.html")`

    Yet visiting `http://127.0.0.1:8000/challenges/january/` threw:

    ```
    TemplateDoesNotExist: challenges/challenge.html
    ```

    ### Investigation

    The error revealed that Django only searched inside built-in apps like `admin` and `auth`, completely ignoring the `challenges` app — even though everything was set up correctly.

    ### The Fix

    The solution was unexpectedly simple:

    ```bash
    CTRL + C
    python manage.py runserver
    ```

    After restarting the development server, the template loaded perfectly.

    ### What Actually Happened

    Django's development server behaves differently depending on what changes:

    | Change Type | Auto-Reload |
    |-------------|------------|
    | Python file changes (`.py`) | ✅ Yes, automatically |
    | New folders or structural changes | ❌ No, requires manual restart |

    Since a new `templates/` directory was added, Django's file watcher didn't pick it up until the server was restarted.

    ### The Lesson

    Sometimes the issue is not in your code — it's in the environment state. Before diving deep into debugging, follow this order:

    <Steps>

        ### Restart the server

        Clear any stale state first.

        ### Recheck the error

        Read the message carefully — Django's error output is informative.

        ### Then investigate

        Dig into configuration and code only after the above steps.

    </Steps>

    <Callout type="info" emoji="💡">
        **Pro Tip:** Make this a habit — *"If something should work but doesn't, restart first."* It can save a surprising amount of time and prevent unnecessary changes to code that was never broken.
    </Callout>

    ## Summary

    | Concept | Key Takeaway |
    |---------|-------------|
    | Templates | Standalone HTML files with optional dynamic content — far better than inline HTML strings |
    | File structure | Place templates in `<app>/templates/<app>/` |
        | `render_to_string` | Converts a template file to a string for use in HTTP responses |
        | `APP_DIRS: True` | Enables automatic template discovery inside registered apps |
        | `INSTALLED_APPS` | Must include your app for `APP_DIRS` to find its templates |
        | Server restart | Required after adding new folders — the dev server doesn't auto-detect structural changes |
        | Static files | CSS, JavaScript, and images — covered in the next section |

        <Callout type="info">
            **Best Practice:** Always use `APP_DIRS: True` for app-specific templates and register your app in `INSTALLED_APPS`. Reserve `DIRS` only for truly global or shared templates.
        </Callout>
